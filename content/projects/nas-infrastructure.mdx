Small businesses usually cannot afford managed cloud hosting for every service, and they
cannot afford specialist infrastructure engineers either. The constraint is that whatever gets
built has to stay up with minimal intervention and be understandable by whoever is next in
the room.

This project built the NAS and hosting infrastructure for a small business client: redundant
file storage, remote access for staff, and WordPress site hosting — all running on a Synology
NAS rather than a rented server farm.

## Storage and redundancy

The Synology NAS was configured with a RAID volume to protect against single-disk failure.
File shares were set up for different access tiers — some shared across the organisation,
others restricted to specific staff roles — with Synology DSM handling permissions and
access control.

Remote access was enabled through Synology's QuickConnect service and a VPN configuration,
giving authorised staff consistent access to shared files from outside the office without
exposing the NAS management interface to the open internet.

## Hosting and reverse proxy

The NAS also serves as the web hosting stack. WordPress sites run behind Synology's built-in
reverse proxy, which handles HTTPS termination with Let's Encrypt certificates. The reverse
proxy routes inbound traffic to the correct virtual host based on the domain name, so
multiple sites can share a single public IP address.

DNS was configured at the registrar level to point each domain at the NAS's public IP, with
Let's Encrypt providing automatic certificate renewal so SSL does not require manual
maintenance.

## Design decisions

The core decision was to centralise services on the NAS rather than split them across a NAS
and a separate hosted VPS. This has a cost advantage but it ties file storage and web hosting
to the same machine — a disk failure during RAID rebuild temporarily affects both. For the
client's workload and risk tolerance that trade-off was acceptable.

The second decision was WordPress for all public sites. WordPress is not the most performant
option, but it is the one that a non-technical employee can update without a developer. That
maintainability constraint drove the technology choice more than any technical consideration.

## What this demonstrates

The project covers the end-to-end setup: DNS, TLS, reverse proxy, file-share permissions,
remote access, and application hosting. None of these pieces is especially complex in
isolation; the value is in configuring them together so they interact correctly and stay
maintainable by someone who was not involved in the original setup.
