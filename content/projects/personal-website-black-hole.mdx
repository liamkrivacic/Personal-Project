This portfolio had to do two things that pull in opposite directions. It has to present the work
clearly, which argues for simplicity, and it has to *be* a piece of work itself, proof of how I
build software, which argues for ambition and taste. The easy move is to pick one. I wanted to see
if I could hold both at once, and this site is the result of refusing to choose.

What came out is a single-page scroll experience that starts as an interactive visual interface
and resolves into a structured project catalogue, backed by real case-study pages, a typed and
validated data layer, and a test suite that covers everything from unit behaviour to full
end-to-end navigation. The page you're reading right now is one of those case studies.

## Architecture

I built the app on **Next.js 16 with the App Router**, React 19, TypeScript in strict mode, and
Tailwind CSS 3. Every piece of project content comes from one typed data file
([src/data/projects.ts](src/data/projects.ts)) and the MDX case-study files in
`content/projects/`, so the content is data, not markup buried in components. Static params are
generated at build time, which means every case-study page ships as pre-rendered HTML.

Fonts are self-hosted through `next/font/google` (Archivo for display, Inter for body,
JetBrains Mono for code), and images are served in AVIF/WebP through `next/image` with Sharp
conversion during the asset pipeline.

## The scroll journey

The heart of it lives in [src/components/scroll-journey.tsx](src/components/scroll-journey.tsx).
A `useEffect` loop takes a smoothed virtual scroll position and drives a handful of CSS custom
properties (`--dive`, `--veil`, `--bg-fade`, `--reveal-col`, `--reveal-list`). Everything that
moves on the page (the black-hole zoom, the veil fade, the heading reveal, the staggered
project cards) is just CSS reacting to those few numbers. I like this approach because the
animation logic stays in one place and the browser does the heavy lifting.

The black hole itself runs in a sandboxed `<iframe>` that I feed scroll progress over
`postMessage`, and the cursor-lighting effect rides in a separate canvas overlay injected once
per page load. Keeping them isolated means one can break or get swapped out without taking the
rest of the page down with it.

## WebGL shader

The black hole is a custom WebGL fragment shader I wrote that combines gravitational lensing
distortion, an accretion disk, and per-frame "heat" that follows the cursor. It lives in
`public/black-hole-tsbxw3/` as a fully self-contained page, so I can iterate on the shader or
replace it entirely without rebuilding the main app bundle, the visual centrepiece is
deliberately the most decoupled thing in the project.

On top of that sits a separate cursor-streamlets layer
(`public/black-hole-cursor-streamlets/`), a small fluid simulation that chases the pointer. I
built it to be compositionally independent from the shader, so the two effects can coexist
without either knowing the other exists.

## Testing

I treated this like production code, which to me means it has tests. The suite covers three
layers:

- **Vitest unit tests** verify the data layer (every project has a slug and valid `imgAlt`;
  image files exist), the case-study frontmatter (required fields, valid dates, slug-to-
  project matching), and source invariants in the component files (no empty alt attributes,
  correct image sizing attributes).
- **Playwright end-to-end tests** verify the scroll journey in a real browser: that the
  black-hole renders, that project cards appear after scrolling, that filters work, and that
  the back-navigation from a case-study page lands on the revealed projects view rather than
  mid-dive.
- **Layout source tests** read raw component source files and assert structural invariants
  that should never regress regardless of refactoring.

## Working with AI

I built this site with Claude as a collaborator, and it's worth being direct about what that
looked like, because I think how you use AI says something about how you think about engineering.

The pattern I settled on: a more capable model acts as a project advisor at decision points
(reviewing architecture, flagging tradeoffs, catching assumptions before they become bugs), while
cheaper subagents handle bounded, well-defined tasks like searching the codebase, checking for
regressions, or drafting content. This mirrors how senior engineers work in teams: expensive
judgment on design decisions, cheaper execution on well-specified work. Getting that split right
required me to understand the work well enough to spec it accurately, which meant I was always
the one who had to hold the system-level view.

The result is that I learned this stack faster than I would have otherwise, but the decisions are
mine. When the shader broke on back-navigation, or the MDX pipeline silently dropped tables, or
the Vercel alias didn't update automatically: diagnosing those required understanding the
architecture, not just prompting. The AI made me faster; it didn't make the thinking unnecessary.

## What this demonstrates

The best part of making my portfolio a real software project is the honesty of the feedback loop:
if my architecture is good, adding a new case study is trivial; if it's fragile, the test suite
tells me before a recruiter ever sees it. The MDX case-study system, the typed data layer, and the
scroll-animation architecture are the decisions that matter in actual production work, not the
ones a university rubric asks for. This site is me showing, rather than claiming, that I can
make those decisions.
